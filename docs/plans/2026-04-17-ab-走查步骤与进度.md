# A/B 批 DevTools 走查步骤与进度

## 1. 用途

这份文档只服务两件事：

1. 把 A、B 两批当前需要人工走查的步骤收口到一个地方。
2. 让走查时可以直接在本文更新状态，再回写总看板。

当前基线：

| 批次 | 当前代码状态 | 当前走查状态 | 当前统计 | 备注 |
| --- | --- | --- | --- | --- |
| A | 已接入 | `未走查` | `0/12 已完成`，`11` 进行中，`1` 待接入 | `采购建议` 仍待切 `news/page`，不在本轮走查通过范围内 |
| B | 已接入 | `未走查` | `0/7 已完成`，`7` 进行中 | 代码已落库并通过自动化验证，尚未做 DevTools 人工走查 |

状态建议统一写法：

| 状态 | 含义 |
| --- | --- |
| `未开始` | 还没走查 |
| `进行中` | 已开始，但还没拿到稳定结论 |
| `通过` | 该检查点已通过 |
| `失败` | 已复现问题，需修复后重走 |
| `阻塞` | 当前受 token、环境、后端数据或接口异常影响，暂时无法判定 |

## 2. 统一前置

构建与环境：

1. 执行 `npm run build:miniapp`。
2. 微信开发者工具打开 `dist/`。
3. 打开 DevTools `Network` 面板，并勾选“保留日志”。
4. 若出现域名或证书校验问题，先启用“开发环境不校验请求域名、TLS 版本及 HTTPS 证书”。

DevTools Console 建议先执行：

```js
wx.setStorageSync("constructmarket_api_mode", "hybrid");
wx.setStorageSync("constructmarket_dev_token", "your-test-token");
wx.setStorageSync("constructmarket_api_base_url", "http://106.15.108.65:8085/api");
getApp().refreshRuntimeConfig();
```

补充说明：

- A 批大部分公共接口可在无 token 情况下验证，但仍建议统一带开发 token。
- B 批会命中多个 `/user/*` 与真实商品详情接口；没有开发 token 时，`product/detail` 可能 `401` 并 fallback，这种情况不算“通过”。
- 若切换了 `API_MODE`、`token` 或 `baseUrl`，都要重新执行一次 `getApp().refreshRuntimeConfig()`。

建议记录口径：

1. 先更新本文的“走查状态”与“结果记录”。
2. 再回写 [`docs/swagger-app-接口映射.md`](/Users/casper/Documents/self/ConstructMarket/docs/swagger-app-接口映射.md)。
3. 最后在 [`progress.md`](/Users/casper/Documents/self/ConstructMarket/progress.md) 追加本次走查结果。

### 3.2 走查步骤

首页：

1. 进入 `/pages/home/index`。
2. 确认页面正常渲染，不是旧 mock 的固定内容。
3. 确认 `Network` 中至少出现并返回成功：
   - `GET /v1/app/home/banners`
   - `GET /v1/app/home/categories`
   - `GET /v1/app/home/new-arrival-products`
   - `GET /v1/app/home/hot-recommend-products`
   - `GET /v1/app/home/news-articles`
4. 点击一个热词，确认跳转到搜索结果页且关键词回填。
5. 点击一个分类卡片，确认可进入选型页。
6. 点击一个商品卡片，确认可进入商品详情页，说明真实 `productId` 已透传。

选型页：

1. 从首页分类卡进入，或直接打开 `/pages/category/index?categoryId=1`。
2. 确认 `Network` 中出现：
   - `GET /v1/app/home/categories`
   - `POST /v1/app/home/search-products?...categoryId=<当前类目ID>&sortType=sales_desc&pageIndex=1&pageSize=6`
3. 切换左侧一级类目，确认右侧子类目和“当前热销”一起刷新。
4. 点击一个二级类目，确认跳到搜索结果页，并带上 `categoryId` 与 `keyword`。
5. 记录一个事实：当前“采购建议”卡仍未切 `news/page`，它不是本轮走查通过项。

搜索结果页：

1. 直接打开 `/package-catalog/search/result?keyword=电钻&categoryId=1`。
2. 确认 `Network` 中出现：
   - `GET /v1/app/home/categories`
   - `GET /v1/app/dict/simple-list`
   - `GET /v1/app/dict/tree-list`
   - `POST /v1/app/home/search-products?...keyword=电钻&categoryId=1&pageIndex=1&pageSize=20`
3. 确认关键词回填、类目 chip、更多类目抽屉、商品结果列表都正常显示。
4. 依次切换“销量 / 价格升序 / 价格降序”，确认每次都重新请求 `search-products`，且 `sortType` 变化正确。
5. 打开筛选抽屉，只选价格区间，确认 `Network` 带上 `minPrice` / `maxPrice`，且结果数量变化。
6. 再选 `material` 或 `minOrder`，确认列表变化即可；当前实现允许这两项只在前端对真实结果做兼容细筛，不要求 `Network` 一定出现对应参数。
7. 打开“查看更多类目”抽屉，切换类目后确认重新请求搜索接口，抽屉关闭，列表刷新。

失败态：

1. 将 `constructmarket_api_base_url` 临时改成错误地址并执行 `getApp().refreshRuntimeConfig()`。
2. 分别重进首页、选型页、搜索结果页。
3. 确认三页都进入 `error` 态，而不是白屏或无限 loading。
4. 用 `?state=offline` 打开三页，确认 `offline` 占位仍然存在。

### 4.2 走查步骤

商品详情基础链路：

1. 从首页、选型页或搜索结果页进入任一商品详情。
2. 确认 `Network` 中出现并成功返回：
   - `GET /v1/app/product/detail`
   - `GET /v1/app/product/specs`
   - `GET /v1/app/merchant/detail?id=<merchantId>`
3. 确认商品标题、价格、主图、SKU 选项、规格说明、店铺卡都来自真实接口，不是固定 mock 文案。
4. 切换至少一组规格，确认 SKU 选中态、价格或库存展示不会错乱。

收藏联动：

1. 在详情页点收藏，确认请求命中 `POST /v1/app/user/favorite/product-favorites/{productId}`。
2. 进入收藏夹，确认该商品出现。
3. 在收藏夹取消收藏，确认请求命中 `DELETE /v1/app/user/favorite/product-favorites/{productId}`。
4. 再回到详情页或搜索结果页，确认收藏态同步变化。

加购联动：

1. 在详情页点击“加入购物车”，确认请求命中 `POST /v1/app/user/cart`。
2. 记录请求体至少包含 `merchantId`、`productId`、`skuCode`、`quantity`。
3. 确认加入成功后，购物车角标或后续购物车页能看到新增商品。
4. 进入收藏夹做一次“快捷加购”，确认同样命中真实购物车接口，且结果联动一致。

浏览与咨询：

1. 首次进入商品详情时，确认请求命中 `POST /v1/app/user/browse/{productId}`。
2. 从详情页进入在线咨询。
3. 发送一条留言，确认请求命中 `POST /v1/app/consult-messages`。
4. 当前只要求“留言成功提交”；不要求完整会话历史、客服回复或实时消息能力。

异常态：

1. 去掉 `constructmarket_dev_token` 后重进商品详情，观察是否出现 `401` 或 fallback。
2. 这种情况只记录为“阻塞/鉴权缺失”，不要判为通过。
3. 若接口报错但页面仍白屏、死循环 loading 或收藏/加购状态错乱，记为失败。
